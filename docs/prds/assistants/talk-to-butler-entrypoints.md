# PRD：把「AionUi 管家」做成各功能的对话入口

## 背景与改动原因

AionUi 内置了一个 **AionUi 管家** 助手，它能用自然语言帮用户：配置助手 / 技能 / MCP / LLM 模型、设置远程访问、诊断各种问题。但**用户根本不知道管家能做这么多**——这些能力藏在一个助手里，没人会主动去用。

参考其他产品的做法：在用户**手动配置的地方**，并排提供一个「用对话来做」的选项。用户点了之后，自动跳回首页、选中 AionUi 管家、并把对应的提示词预填进输入框——用户直接回车就能让管家帮他完成这件事。

**目标**：把管家的能力，分散嵌入到它本该出现的每一个场景里，让用户在需要时自然发现并用上它。

---

## 文案规范（全站统一）

入口采用「对话 / 手动」对仗结构，动词随场景变化：

| 场景类型                         | 英文                              | 中文                        |
| -------------------------------- | --------------------------------- | --------------------------- |
| 创建（助手 / Agent）             | Create via chat / Create manually | **对话创建** / **手动创建** |
| 添加（模型 / MCP / 技能 / 远程） | Add via chat / Add manually       | **对话添加** / **手动添加** |
| 排查（上报问题）                 | Solve via chat / Solve manually   | **对话排查** / **手动排查** |

---

## 通用交互

**入口形态**：参考产品的 split button（主按钮 + 下拉箭头），下拉两项：「对话{动词}」在上、「手动{动词}」在下，各带图标。

- 已经是下拉菜单的入口（如 MCP 添加），直接在菜单里追加「对话添加」一项即可，不必改成 split button。
- 上报问题弹窗是特例：保持「提交」按钮，旁边**并排**加一个「对话排查」按钮（两条不同的路：一条提交给官方、一条自己让管家解决）。

**点击「对话{动词}」后**：

1. 跳回首页；
2. 自动选中 AionUi 管家；
3. 把该场景对应的提示词预填进输入框并聚焦；
4. **不自动发送**——让用户能看一眼、按需修改后再发。

**边界规则（管家被禁用时）**：管家是内置助手，用户可以手动关掉它。若用户点了「对话{动词}」时管家处于禁用状态，**自动帮用户重新启用它**，再跳转，并用一句轻提示告知「已为你启用 AionUi 管家」。理由：用户点这个按钮本身就是想用它，自动启用最顺手，符合管家「以用户方便为主」的定位。

## 专项：上报问题 →「对话排查」

**改动原因**：用户上报问题后，要等官方处理。其实很多问题管家当场就能诊断。给上报弹窗加一条「自己先让管家排查」的快捷路径。

**期望效果**：上报弹窗里，「提交」按钮旁并排一个「对话排查」。点击后：

1. 把用户已填的**问题描述** + **所选模块** + 已采集的环境信息（版本 / 系统等），拼成一段**结构化提示词**；
2. 把用户上传 / 自动截取的**截图**一并带入首页输入框（复用现有的「粘贴文件进输入框」通道）；
3. 跳回首页、选中管家、内容已结构化填好 —— 用户直接发送，管家即可开始排查。

**结构化提示词示意**：

```
我在使用 AionUi 时遇到问题，帮我排查一下。

【模块】{模块名}
【问题描述】{用户填写的内容}
【附件】见输入框中的截图

请帮我诊断原因并给出解决办法。
```

**是否自动发送**：同样**不自动发**，预填好后让用户确认再发（截图带入需要极短的处理时间，自动发容易抢跑）。

## 范围

一次性铺设全部 7 个接入点（统一文案、统一交互机制后一起做），不分批。
