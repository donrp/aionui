# 二期 · Agent 统一为助手

> 本期把底层的 Agent 纳入助手体系：Agent 页变成纯体检台，每个可用 Agent 自动生成一个开箱即用的"裸助手"，首页与团队的选择统一收敛到"助手"这一个概念。
>
> 概念、术语、整体路线见[总纲](./README.md)。本期复用[一期](./phase-1-assistant-governance.md)定稿的助手配置结构与默认值行为。

---

## 一、要解决的问题

一期把"助手"做厚了，但围绕"用什么发起对话"，产品里仍并存两套割裂的东西：

- **Agent 页**只告诉用户"检测到哪些 CLI"，不告诉"能不能用"——很多 Agent 要到真正发消息时才暴露连接失败、未登录，用户措手不及。
- **首页**同时摆着"直接选 Agent"（顶部 pill）和"选助手"（下方区域）两套入口，用户分不清自己在选引擎还是选预设；而且"直接选的 Agent"是个没有任何配置的裸引擎，享受不到一期做厚的助手能力。
- **团队**选队友时，列表里把"引擎"和"助手"混在一起，同样割裂。

## 二、目标

- **Agent 页成为可信的体检台**：不仅说"找到了什么"，还说"能不能连上、怎么测"。
- **开箱即用**：每个可用 Agent 自动生成一个"裸助手"，它就是一种普通助手，复用一期的全部配置能力。
- **首页只摆助手**：取消"直接选 Agent"入口，用户面对的只有助手，点中即用。
- **团队选队友 = 选助手**：与首页同源。

## 三、本期不做（Non-Goals）

| 不做的事 | 留给 / 说明 |
| --- | --- |
| 助手配置项、默认值两态行为 | 一期已完成，本期直接复用 |
| 内置助手 / 官方技能的热更 | 三期 |
| 重做"记住上次"偏好逻辑 | 一期已按助手实现；裸助手自然继承，无需新开发 |

## 四、功能需求

### F2-01 Agent 页：检测与可用性展示

**现状**：「设置 → Agent」页（本地 tab）以卡片网格列出检测到的 Agent，只反映"是否在系统中找到"，不反映"能否真正连上"。

**需求**：每个 Agent 卡片展示图标、名称，以及三选一的**可用性状态**：

| 状态 | 含义 |
| --- | --- |
| **可用** | 已找到，且能成功建立连接 |
| **不可用** | 已找到，但连接失败 |
| **未找到** | 电脑上没有这个工具 |

- 「不可用」时，在状态旁放一个信息标记（ⓘ），失败原因与解决指引收在标记里，**hover 才展开**，不长期占用版面。
- 本页是纯运维体检台——只回答"装了哪些、能不能用、怎么修"，不发起对话、不配规则/技能/MCP/模型。

> "可用"的判定做到**连接级**：不仅确认程序装好，还实际尝试连一次、确认能连通。更深的问题（未登录、额度不足）由用户真正发起对话时由对话界面提示，不在本页探测。

**验收**：
- [ ] 每个 Agent 区分"可用 / 不可用 / 未找到"三态
- [ ] 状态反映"能否连接"而非仅"是否装好"
- [ ] 「不可用」的原因与指引收在状态旁 ⓘ 标记内，hover 展开
- [ ] 本页无发起对话或配置规则/技能的入口

---

### F2-02 测试连接

**需求**：每个 Agent 卡片都有**「测试连接」按钮**——无论当前可用、不可用还是未找到，都能点。

- 点击后进入测试中状态（loading）。
- 测试完成即时刷新可用性状态。
- 若由不可用恢复为可用，对应的裸助手随之恢复可用（见 F2-04）。

**验收**：
- [ ] 每个 Agent 都有「测试连接」按钮（三态一律都有）
- [ ] 测试中有明确进行中反馈
- [ ] 测试结果实时刷新可用性状态

---

### F2-03 自定义 Agent 识别

**现状**：「设置 → Agent」页已支持添加自定义 Agent（填写名称、启动命令、参数、环境变量等）。这些配置较专业，用户常常难以独立填对。

**需求**：沿用现有自定义 Agent 表单，并在**表单顶部固定一条引导**：不确定怎么填时，「去问 AionCLI」——把要接入的工具名告诉它，让它帮忙查清启动命令、参数怎么配。提供一个跳去 AionCLI 对话的快捷入口。

> AionCLI 是内置、随应用自带、始终可用的 Agent，用它当"配置顾问"比让用户翻文档/找客服更顺手。

- 保存后，自定义 Agent 进入与内置 Agent 一致的检测流程，检测可用后同样自动生成裸助手。

**验收**：
- [ ] 自定义 Agent 表单顶部有「去问 AionCLI 帮忙配置」的引导与快捷入口
- [ ] 自定义 Agent 与内置 Agent 在检测、裸助手生成上行为一致

---

### F2-04 裸助手自动生成与生命周期

**需求**：当一个 Agent 被检测为可用，系统自动在助手列表中为它生成一个**裸助手**——一种开箱即用的普通助手，名称/头像取自该 Agent，规则等留空，复用一期定稿的全部助手配置结构。

- 裸助手出现在首页，用户可直接选中发起对话，无需任何配置。
- **一个 Agent 仅自动生成一个裸助手。** 用户若想要基于同一 Agent 的多个助手，自行复制或新建（成为自定义助手）。
- 裸助手可被就地编辑（它是普通助手）；编辑后直接展示新值，系统不保留"原始裸助手"副本。

**裸助手不可删除**：裸助手是系统对"这个 Agent 可用"的呈现，与 Agent 绑定，不是用户的创作物，因此不提供删除。想让它从首页隐藏，用助手开关关掉即可（见一期 F1-08）。删除只针对用户自己创建的自定义助手。

> 只有用户拥有的助手（自定义助手）可删除；系统/官方拥有的助手（裸助手、内置助手）都不可删除。

**Agent 可用性变化时的联动**：

- Agent 从可用变为不可用：对应裸助手**不消失、不删除**（可能挂有历史对话），仅给出提示（见 F2-05）。
- Agent 恢复可用：裸助手自动恢复可用，无需用户操作。

**验收**：
- [ ] 每个可用 Agent 自动生成且仅生成一个裸助手
- [ ] 裸助手可直接发起对话，也可像普通助手一样编辑，但不可删除
- [ ] 想隐藏裸助手只能用开关关闭，不提供删除
- [ ] Agent 不可用时裸助手保留并提示，不删除；恢复后自动恢复可用

### F2-05 Agent 不可用提示

**需求**：当助手依赖的 Agent 检测为不可用时，在助手相关界面里，该助手**名称旁显示一个感叹号标记**，hover 弹出"Agent 不可用，请检查"。不长期占用列表空间，也不自动禁用或删除助手。

**验收**：
- [ ] Agent 不可用时，对应助手名称旁出现感叹号标记
- [ ] hover 标记弹出原因提示
- [ ] 提示不影响助手的存在与历史对话

---

### F2-06 首页：统一为单一助手选择栏

**现状**：首页（GUID 页）有两套并存的选择界面——顶部一排 pill 直接选 Agent，下方一个区域选助手。选中助手后还会进入一个二级界面（含返回、助手名、切换 Agent 等）。

**需求**：把两套合并成**一排横向的助手 pill**，首页只摆助手：

- pill 里展示所有开启状态的助手（裸 / 内置 / 自定义混排，不分组、不标注来源），顺序沿用助手排序（一期 F1-09）。
- 点击某个助手 pill 即选中，下方立即展示该助手的推荐提示词（见 F2-08），输入框就绪，可直接输入发送。
- **取消"直接选 Agent"入口**——裸助手已经代表了对应 Agent，不再单独列引擎。
- **取消选中助手后的二级界面**，也取消 pill 旁的"切换 Agent"下拉——换助手就在 pill 栏点一下完成；想换引擎，就是换一个助手。

**验收**：
- [ ] 首页只剩助手一种可选对象，原两套界面合并为单一助手 pill 栏
- [ ] 点击助手 pill 即选中并就绪，不进入二级页
- [ ] pill 顺序遵循助手排序
- [ ] 不再有"直接选 Agent"入口与"切换 Agent"下拉

---

### F2-07 首页：助手过多时的收纳

**需求**：当助手数量超出一排可容纳的范围：

- pill 栏横向展示前 N 个助手。
- pill 栏**最右侧**有一个下拉按钮（「⌄」），点击在其下方弹出完整助手列表（带搜索）供浏览和选中。
- 桌面端与移动端形态一致，避免移动端纯横滑找不到目标。

**验收**：
- [ ] pill 栏展示前 N 个助手
- [ ] 最右侧下拉按钮可展开完整助手列表并选中
- [ ] 桌面端与移动端形态一致

---

### F2-08 首页：选中态持久化、默认选中与推荐提示词

**需求**：

- **持久化**：用户上次选中的助手，下次进入首页保持选中。
- **新用户默认**：全新用户（还没选过任何助手）默认选中 **AionCLI 对应的裸助手**——AionCLI 内置、始终可用，作为默认落点保证总有一个助手能直接开聊。
- **推荐提示词区**：选中助手后，输入框下方展示该助手的推荐提示词，点击即填充输入框。

**验收**：
- [ ] 上次选中的助手下次进入首页保持选中
- [ ] 全新用户默认选中 AionCLI 裸助手
- [ ] 选中助手后展示其推荐提示词，点击填充输入框

---

### F2-09 团队：选队友 = 从可用助手中选

**现状**：团队创建/添加成员时，候选列表把 Agent（引擎）和助手混在一起列。底层成员记录已能关联到一个助手并继承其配置。

**需求**：组队挑选 Leader 与成员时，从**与首页同一批助手**里挑（裸 / 内置 / 自定义都行），不再出现"光秃秃的引擎"项。

- 列表与首页同源、长相一致（每项就是头像 + 名称），**不向用户标注助手类型**。
- 列表同样受助手开关（关闭则不出现，见一期 F1-08）与助手排序（见一期 F1-09）的影响。
- 选中某助手作为成员后，该成员继承该助手的全部配置（规则、技能、默认模型/权限等）。
- 产品目标是所有助手都能当 Leader 或成员。

**有些助手暂时不能进团队**：团队协作需要 Agent 支持一种特定的通信方式，并非所有 Agent 都支持。不支持的 Agent，其助手在候选列表里**显示为灰色、不可选，并附一句简短说明**（如"该 Agent 暂不支持团队协作"），而不是从列表里藏掉——让用户知道它存在、只是暂时不可用于团队。这是底层技术限制，不是把助手分等级。

**验收**：
- [ ] 组队候选列表与首页助手同源、长相一致，不标注类型
- [ ] 裸 / 内置 / 自定义助手都能作为成员候选
- [ ] 候选列表受助手开关与排序影响
- [ ] 选中助手作为成员后继承其配置
- [ ] 不支持团队的助手灰显并说明原因，而非隐藏

## 五、衔接说明

- 本期复用一期的助手配置结构与默认值两态行为——裸助手作为一种普通助手，自动获得这些能力，无需额外开发。
- 一期就绪的助手开关、排序，本期延续到首页 pill 栏与团队候选列表。
- 内置助手在本期仍随应用版本更新；三期把它升级为热更。

## 六、成功标准

- 用户在 Agent 页能看清每个 Agent 能不能用、并能手动测试连接。
- 装好一个 CLI 即得到一个开箱即用的助手，无需配置即可对话。
- 首页只有"助手"一种选择，点中即用、无二级页。
- 组队时从同一批助手里挑队友。
