# 笔记本(记忆系统) — 脑暴沉淀

> 状态:脑暴结论,非完整 PRD。来源于用户洞察(用户在同一个工作台处理多种活,希望记忆尽可能通用)。
> 用途:锁定已拍板的设计决策,作为后续写正式 PRD 的输入。

## 一句话

给 AionUi 的助手一个「笔记本」:agent 在对话中实时记录值得长期保留的事实,
新对话自动读回这些笔记,让用户"换 CLI 不失忆、换对话不失忆"。

## 这个需求要回答的问题

AionUi 现状是「记忆孤岛」:

- 换 CLI 失忆:用户在 Claude 沉淀的 `CLAUDE.md`,Gemini / Qwen 完全看不到,各 CLI 各管各的记忆文件。
- 换对话失忆:同一个 CLI,新对话也是一张白纸,不记得上一个对话聊过啥。

用户的核心诉求(原话):"agent 万一挂掉了,换个会话还能想起来" —— 本质要的是**别丢**,不是高质量总结。

## 为什么挂在「助手」上

AionUi 里助手是**独立于 CLI 的一层**(同一助手底下可换 Claude / Gemini / Qwen),且**跨会话存在**。
所以记忆挂在助手上,天然同时解决两个痛点:

- 换 CLI 记忆还在(横向打通)
- 换对话记忆还在(纵向沉淀)

现状已有地基:助手配置里有 `context` 字段(承载助手级指令),但目前是用户手填的静态指令,
不是能自动沉淀、累积的记忆。地基有,差的是让它"活"起来。

- 助手类型:`packages/desktop/src/common/types/agent/assistantTypes.ts`

## 已拍板的设计决策

| 维度     | 结论                                                | 备注                                                                |
| -------- | --------------------------------------------------- | ------------------------------------------------------------------- |
| 形态     | 助手详情页里的「笔记本」Tab,一条条带时间戳的笔记    | 线框图已画:身份 / 规则 / 工具 / 技能 / **笔记本**                   |
| 锚点     | 挂在助手上                                          | 助手是独立于 CLI、跨会话的一层                                      |
| 分层     | **两层**:助手级(跨项目)/ 助手 × 项目级(锁项目)      | 砍掉了独立的"全用户层"——对目标用户是伪需求,记在助手层够用           |
| 归类     | **agent 自动**判断记哪一层                          | 用户分不清"助手 vs 项目"层级,自动归类才符合"自动记录"的初衷         |
| 用户操作 | **只看和删**                                        | 不手动记、不手动分层;笔记本是"agent 替你记、你偶尔审一眼"的后台沉淀 |
| 写入时机 | **实时随手记**                                      | 为了"中途挂掉也别丢";对话结束才总结扛不住中途崩溃                   |
| 读取时机 | **新对话自动注入**对应层(助手级 + 当前项目级)的笔记 | "换会话能想起来"必须靠自动读回才成立                                |

## 为什么分两层(以助手为锚)

| 记忆类型                          | 例子                               | 范围                                     |
| --------------------------------- | ---------------------------------- | ---------------------------------------- |
| 助手级(关于你/助手的稳定事实)     | 偏好、编码规范、助手角色           | 跨项目,跟着助手走                        |
| 助手×项目级(关于某摊具体活的事实) | 某项目的状态、约束、关键决策、进展 | 锁在该项目,不跨,避免"办公楼的事跑到别墅" |

类比:Claude Code 的 `~/.claude/CLAUDE.md`(全局)+ 仓库内 `./CLAUDE.md`(项目级)两层。

## 核心机制(读 + 写双向闭环)

```
对话中 ──[agent 实时记,只记稳定事实 + 查重]──> 笔记本(分助手层 / 项目层)
新对话 <──[自动注入对应层的笔记]────────────── 笔记本
用户   ──[随时看 / 删]──> 笔记本
```

## 实时记的两条内部规矩(提示词 + 工具层消化,不暴露给用户)

1. **只记"以后还用得上的稳定事实"**:偏好、规范、项目约束、关键决策。
   不记寒暄、一次性问答、过程性废话 —— 否则笔记本三天变垃圾场,"看和删"负担比不记还重。
2. **写前查重**:已记过的别记第二遍;同一事实更新而非新增。

## 实现形态(待技术确认,倾向)

倾向与 Kanban 同路子:做成 agent 能调的一套工具(write_note / list_notes / delete_note),
"什么时机记"最终落成"提示词怎么约束 agent 调这套工具"。具体交由技术决定。

## 待讨论 / 未拍板

- 自动注入笔记时,如果笔记很多,会不会撑爆上下文?是否需要"注入时按相关性/时间裁剪"(交由技术评估)。
- agent 自动归层偶尔归错,用户除了"删",要不要给一个"移动到另一层"的轻量操作?(当前定为只看和删,先不加)
- 笔记本与各 CLI 原生记忆文件(CLAUDE.md / GEMINI.md)的关系:是完全替代,还是 AionUi 笔记本作为统一源、按需生成各 CLI 的文件?(影响"跨 CLI 通用"的彻底程度,留待技术调研)
- 写入时机虽定为"实时",是否需要一个轻量的"暂存→稍后归档"以进一步降噪?(当前先用"实时 + 两条规矩",观察噪音再说)
