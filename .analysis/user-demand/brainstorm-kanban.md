# Kanban 看板 — 脑暴沉淀

> 状态:脑暴结论,非完整 PRD。来源于用户洞察(非典型开发用户需要清晰的任务进度跟踪)。
> 用途:锁定已拍板的设计决策,作为后续写正式 PRD 的输入。

## 一句话

给 AionUi 用户一块「工作区级的任务看板」,人和 AI agent 共同维护,用来跟踪"在同一个工作台里处理的多种活"的整体进度。

## 这个需求要回答的问题

用户(尤其是非典型开发用户)在 AionUi 里跨多个对话、做多种活(写代码 + 文档 + 运维 + 内容…),
缺一个地方看到"我这摊事整体到哪了"。进度被一条条对话切碎,没有全局视角。

## 已拍板的设计决策

| 维度       | 结论                                                                         | 备注                                                                                                                                                 |
| ---------- | ---------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| 实现形态   | MCP server(`mcp-kanban`),agent 通过 create / edit / move issue 等工具操作    | 选 MCP 而非扩展现有协议,因为 MCP 是所有主流 CLI(Claude / Gemini / Qwen / iFlow)的通用接口,一套工具适配所有 agent,不必逐个适配各 CLI 的原生 plan 机制 |
| 层级       | 工作区级,一个工作区一块板                                                    | 用户洞察、线框图、现有数据结构三者都指向工作区级;一个工作区一块板 → agent 无需选择往哪写,化解归属难题                                                |
| 入口       | 对话页右侧面板的第三个 Tab「任务」,与「文件 / 变更」并列                     | 贴合"在同一个地方处理多种活"的洞察;改动最轻(右侧面板已有 Tab 机制);需标明该板为当前工作区共享                                                        |
| 模式开关   | 工作区级开关。开 = agent 拿到看板工具 + 提示词告知该拆任务上板;关 = 收走工具 | 与用户最初"启动 kanban 模式"的设想一致;避免 agent 在简单问答时也乱建一堆卡                                                                           |
| Tab 常驻   | 「任务」Tab 始终常驻;未开模式时显示空状态 + "开启看板模式"引导               | 目标用户为非典型开发者,功能藏太深发现不了;入口需自解释                                                                                               |
| 关模式行为 | 数据不删,仅 agent 不再碰板;用户随时可再开,数据仍在                           | 关模式 ≠ 删数据                                                                                                                                      |
| 共管       | 人和 agent 都能建 / 改 / 拖卡                                                |                                                                                                                                                      |
| 卡片归属   | 记 creator(谁建的:用户 / 哪个 agent),不记来源会话                            | 来源会话会过期(对话可删可归档),且增加耦合;creator 稳定、天然兼容多 agent、让用户一眼看出哪些是 AI 自作主张建的                                       |

## 看板形态(来自线框图)

- 列 = 状态:Backlog / Todo / In Progress / Done
- 卡片 = issue:编号(TASK-N)+ 标题 + 描述 + 优先级
- 「任务」Tab 与「聊天」并列在工作区维度

## 卡片字段

| 字段        | 含义                                | 例                           |
| ----------- | ----------------------------------- | ---------------------------- |
| 标题 / 描述 |                                     |                              |
| 状态        | Backlog / Todo / In Progress / Done |                              |
| 优先级      | Low / Medium / High                 |                              |
| **creator** | 谁建的这张卡                        | `user` / `claude` / `gemini` |
| owner       | 这张卡归谁干(Team 场景尤其有用)     | 某 agent                     |
| 依赖        | blocked_by / blocks                 | 先不暴露给 UI,数据层保留     |

## 与现状的衔接

- **复用并泛化现有 `team_tasks` 表**:该表已有 status / owner / blocked_by / blocks / metadata 等字段,
  结构与看板卡片几乎一致。让单人工作区也能用这张表,避免项目里出现两套任务系统。
  - 数据层位置:`packages/desktop/src/process/services/database/schema.ts`
- 现有的「单 agent 临时计划」(`MessagePlan.tsx`,跟着消息走、会话关了就散)和本看板是两回事;
  本看板要的是持久、独立、人机共管的工作区级任务管理。
- 现有「定时任务页(Cron)」是项目里唯一「独立 + 持久 + 用户可管理」的任务列表范式,可参考其交互。

## 待讨论 / 未拍板
