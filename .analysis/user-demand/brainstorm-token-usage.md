# Token 用量统计 — 已洞察 · 暂缓(不做)

> 状态:**暂缓,当前不做**。来源于用户洞察(用户对成本/用量敏感)。
> 用途:保留洞察与暂缓的论证,待根因消除后可直接接续。

## 洞察(为什么曾想做)

用户访谈/数据显示:AionUi 用户对成本敏感,平均绑定 3 个 CLI、多 CLI × 多 LLM 混用。
直觉需求:一个用量面板,让用户"心里有数"——这段时间用了多少、大头在哪。

## 决定:暂缓,当前不做

经一路推演,三条论证指向同一个根因,最终决定不做。

### 论证一:成本($)做不准 → 否决

- 各模型单价不一、且会变;用户还可能用自己的 API key / 第三方中转,实际单价 AionUi 根本不知道。
- 内置单价表会过时、会误导。
- 结论:**不做成本($),只可能做 token 数量**。

### 论证二:多维用量面板(总量 + 按 CLI/模型/项目拆分)→ ROI 不对

- 与 Kanban / 笔记本不同:那两个是"创造一个可掌控的新东西";token 统计是"测量一个不可控的现实"。
- 数据不在 AionUi 手里,在各 CLI 手里,且各家给的不一样:
  | Agent | 能给的 token 数据 |
  |---|---|
  | 本地 Claude Code | input/output 分得清,最全 |
  | ACP 系(Claude/Qwen/iFlow) | 只有一个"上下文总数"黑盒 |
  | Codex | 运行时基本拿不到 |
- 只要底下有个 Codex 给不出数,"总量"就天生残缺;残缺的总量对"心里有数"帮助有限,反而让用户不踏实。
- 给不出数据的 agent 的处理立场曾定为:**显式标注"不支持"**(诚实,不显示 0、不静默忽略)。
- 但即便如此,做一个注定残缺、且要迁就一堆 CLI 差异的面板,换来的价值不刚性。ROI 不对。

### 论证三:连"单会话实时上下文"也算不准 → 退无可退

- 本以为"单会话实时上下文(还剩多少)"是不依赖跨 agent 聚合的安全台阶,但同样不准:
  - ACP 报的"上下文总数"是各 CLI 自己的口径(算不算 system prompt / 工具定义 / 缓存,各家不一),AionUi 拿到的是黑盒数字。
  - Codex 不报。
  - "还剩多少"= 总数 ÷ 上限,而上限(context window)各模型不同、可能动态变,分母也未必拿得准。

## 根因(暂缓的真正原因)

**token 的"真相"不在 AionUi 手里,在各个 CLI 手里,且各家口径不一。**
AionUi 在 token 这件事上自始至终是"拿二手数据的人",算不准是**结构性**的,不是努力能解决的。
在这个根因消除前,做任何 token 统计都是在不准的地基上盖楼。

## 未来重启的前提

若以下任一条件成立,可重新评估:

1. AionUi 有了**统一的 token 采集层**(对各 CLI 运行时统一拦截/归一口径),数据掌握在自己手里。
2. 各 CLI 的 token 上报口径趋于统一,或 AionUi 对其掌控力显著增强。
3. 出现"事后解析各 CLI 本地日志"等可补齐数据的可行路径,且覆盖面足够。

重启时,展示维度的初步想法(供参考):

- 采集天然落在 CLI(只有 CLI 知道烧了多少);
- 展示可双维度:按 CLI(哪个引擎费油)+ 按助手(哪件活烧钱);
- 目标用户(非典型开发者)更关心"助手维度"(哪件活烧钱),会话已存 `preset_assistant_id`,助手归属大概率现成。

## 参考(仅作事实,不影响本决定)

社区已有人尝试过类似功能(走"事后扫描各 CLI 本地日志、只做 token 不做成本"的路线),
印证了"只做 token、不做成本"可行、且各 CLI 本地日志里确有 token 数据可挖。
但本需求重新评估后决定暂缓,不以该尝试为基础。
